# Brotato stat mechanics

Reference facts derived from decompiled code, for judging itemization/stat-priority tradeoffs without re-deriving from scratch:

- **stat_curse**: positive curse scales enemy damage/HP via a `sqrt(curse)` factor (`entity_service.gd`) — avoid. Negative curse is clamped to zero benefit — harmless to have, but not a defensive gain either.
- **Stat caps**: `utils.gd`'s `get_capped_stat`/`get_max_capped_stat` (469-508) match exactly 5 stats: `stat_max_hp`, `stat_speed`, `stat_dodge`, `stat_curse`, and `stat_crit_chance`; every other stat falls through to `printerr("no associated cap")` — i.e. is definitionally uncapped. The default cap *values* matter as much as the list membership (`player_run_data.gd::init_stats`, 430-436): `hp_cap`/`speed_cap`/`crit_chance_cap` default to `LARGE_NUMBER` (99999999) — effectively infinite until an item (e.g. Handcuffs/Shackles) sets a real cap — while `dodge_cap = 60` and `curse_cap = 0` bind from the start of every run.
- **stat_crit_chance**, specifically: because `crit_chance_cap` defaults to `LARGE_NUMBER`, the game-side cap on the player's stat is a no-op in the common case — but the DPS engine (`docs/dps-engine.md`, step C) still clamps the *combined* chance (weapon base + capped player stat/100) to `[0, 1]`, since a probability can't exceed certainty regardless of what the in-game cap allows. Crit is folded into DPS as an expected value — `(1-cc)*d2 + cc*round(d2*crit_damage)` — not rolled or reported as a separate burst line. This is the same `%damage`-bracket-plus-crit interaction described just below for `stat_percent_damage`: both land inside the same per-hit pipeline the engine now replicates exactly.
- **stat_attack_speed** is a universal cooldown-reducing multiplier, confirmed applied identically in both `ranged_shooting_data.gd` and `melee_shooting_data.gd` — never dead weight regardless of weapon type.
- **stat_hp_regeneration** at or below 0 is a harmless no-op (`player.gd`'s `check_hp_regen`/`set_hp_regen_timer_value` just stops the regen timer) — unlike negative lifesteal, which actively drains HP on hit.
- **stat_percent_damage** is a multiplicative global weapon-damage bonus: `weapon_service.gd:239` computes `percent_dmg_bonus = 1 + stat/100`, which is summed with `set_bonus_dmg` and `exploding_dmg_bonus` and applied at `weapon_service.gd:249` as `damage = max(1, round(damage * (set_bonus_dmg + percent_dmg_bonus + exploding_dmg_bonus)))`. Not in utils.gd's cap list — uncapped. Negative values reduce all weapon damage, floored at 1. Applies to held weapons; non-pet structures substitute `structure_percent_damage` instead (`weapon_service.gd:240-243`).
- **stat_armor** reduces damage taken via a diminishing-returns coefficient, `run_data.gd::get_armor_coef` (1429-1439): `percent_dmg_taken = 10 / (10 + abs(armor)/1.5)`, applied in `player.gd::get_damage_value` (310-311) as `max(1, round(dmg_value * armor_coef))` — never reaches 0 (damage floor 1), so armor can only reduce, never eliminate, incoming damage. Not in utils.gd's cap list — uncapped. NEGATIVE armor is actively harmful: the coefficient becomes `(1 - percent_dmg_taken) + 1` (i.e. `2 - coef`), amplifying damage taken above 100%.
- **stat_luck** is a drop-chance multiplier, verified in two places in `item_service.gd`: consumable drops (161-165) use `min(1.0, base_drop_chance * (1 + luck/100))`; item-tier wave drops (523-537) use `wave_chance = wave_base_chance * (1 + luck/100)` when luck >= 0 (the code's `luck` variable is already `stat/100` at line 523; both formulas here are written in raw-stat units). Not in utils.gd's cap list — uncapped. NEGATIVE luck does not just subtract — it DIVIDES: `wave_chance = wave_base_chance / (1 + abs(luck)/100)`, so negative luck is a diminishing-returns penalty rather than a symmetric one.
- **stat_harvesting** pays out at end-of-wave via `main.gd::manage_harvesting()` (1289-1325): `val = harvesting_stat + bonuses (pacifist/cryptid/elite-kill/living-enemy/charmed)`; if `val >= 0`, it grants BOTH `RunData.add_gold(val)` AND `RunData.add_xp(val)` — one point of harvesting is worth one material and one XP, every wave. If `val < 0`, it instead calls `RunData.remove_gold(abs(val))` — negative harvesting actively drains gold each wave, it is not a no-op. Separately, `main.gd::_on_HarvestingTimer_timeout()` (1521-1549) grows the stat itself while `harvesting_stat > 0` and the current wave is within `nb_of_waves`: `val = ceil(harvesting_stat * harvesting_growth/100)` (default `harvesting_growth = 5`, `player_run_data.gd:448`). In endless (wave > nb_of_waves) it instead decays: `val = ceil(harvesting_stat * ENDLESS_HARVESTING_DECREASE/100)` (`ENDLESS_HARVESTING_DECREASE = 20`, `run_data.gd:18`). At `harvesting_stat <= 0` the timeout handler does neither (no growth, no decay). Not in utils.gd's cap list — uncapped.
- **stat_range** adds flat range to weapons: ranged weapons get the full stat via `weapon_service.gd::init_ranged_stats` (42) and `init_ranged_pet_stats` (78) — `max_range = max(min_range, max_range + stat_range)`; melee weapons get only half via `weapon_service.gd::init_melee_stats` (29) and `init_melee_pet_stats` (66) — `max_range = max(min_range, max_range + stat_range/2)`. Both are floored at that weapon's own `min_range`, so it can shrink effective range increases but never below the weapon's floor. Not in utils.gd's cap list — uncapped. Weapons flagged `increase_projectile_speed_with_range` (only Flamethrower T2–T4 ship with this) also get a projectile-speed boost from `stat_range`: `projectile_speed = clamp(projectile_speed + projectile_speed/300 * stat_range, 50, 6000)` (`weapon_service.gd::_set_common_ranged_stats:115`).

## Cap-at-current-value item archetype

Some items grant a strong stat bonus in exchange for permanently freezing a *different* stat at its current value for the rest of the run. Observed on Handcuffs (`key="hp_cap"`, `text_key="EFFECT_HP_CAP_AT_CURRENT_VALUE"`) and Shackles (`key="speed_cap"`, `text_key="EFFECT_SPEED_CAP_AT_CURRENT_VALUE"`). Detect via the `key`/`text_key` naming pattern, not `effect_sign` — that field is inconsistent across these items (Handcuffs uses 0, Shackles uses 3), so it can't be used to spot the archetype. When evaluating such an item, weigh how costly freezing that specific stat is for the build's remaining waves (e.g. freezing Max HP is much worse than freezing Speed for a build that's already capped or doesn't need more).

## Ranger character kit

Reverse-engineered from `items/characters/ranger/ranger_data.tres` + effect files:
- `wanted_tags=["stat_ranged_damage","stat_range"]`, `banned_item_groups=["melee_damage"]`, `no_melee_weapons` effect.
- Flat `+50 stat_range`.
- `+50%` gain modifier on `stat_ranged_damage` (`ranger_effect_4.tres`, key=`effect_increase_stat_gains`).
- `-25%` gain modifier on `stat_max_hp` (`ranger_effect_6.tres`, key=`effect_reduce_stat_gains`).

Gain modifiers are applied multiplicatively to the raw effects-dict sum at *display* time: `displayed_stat = raw_effects_sum * character_gain_modifier`. Verified: raw RD 6 × 1.5 = displayed 9; raw max_hp 48 × 0.75 = displayed 36. Implementation: `recovered/effects/items/stat_gains_modification_effect.gd`.
